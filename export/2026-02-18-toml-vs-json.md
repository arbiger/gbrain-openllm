### Script arguments toml

This type of toml contains the parameters or options that control the training process and its output. Options like learn-rate, rank, alpha, scheduler, optimizer, noise type, file paths, precision, and general dataset configs. This file is autogenerated and can be treated like a temp file.

In the past version of the gui, parameters would've been passed via command-line arguments, meaning this type of toml was not used in version 23.x or earlier. Now as of version 24.x of the gui, this type of toml file is created whenever the gui runs a script or begins training, and contains the arguments for the process that's being run.

The [given reason](https://github.com/bmaltais/kohya_ss/releases/tag/v24.0.1) for the use of toml instead of full command-lines is better security from malicious config arguments.

### Dataset config toml

This type of toml describes how your input files (image and captions) should be handled. A lot of documentation on this type of file is given here: [https://github.com/kohya-ss/sd-scripts/blob/main/docs/config_README-en.md](https://github.com/kohya-ss/sd-scripts/blob/main/docs/config_README-en.md)

Personally I haven't found it necessary to create or edit this type of toml, but it could be useful for cases where you're training multiple concepts with several image folders, and want to override certain options on a specific folder.

### Gui config json

This type of file contains the options that are configurable in the gui. You can think of it as your project file, which you can use to save and load your settings in the gui. The json config is only used by the gui and not passed to the training scripts.

A copy of the config json is autogenerated along with the script arguments toml file when you start training.

### 1. Script arguments toml (腳本參數設定)

- **您的理解：** 這是控制程式運作的核心參數（學習率、路徑、優化器），以前用落落長的指令列 (Command Line)，現在改用 TOML 檔，比較安全且易讀。
    
- **對應到 OpenClaw (Mag)：** 這就是您的 **`~/.openclaw/config.toml`**。
    
    - 它控制了 Mag 的「大腦」是誰 (Ollama/Google)、記憶體在哪 (Local Vector)、以及超時時間 (Timeout)。
        
    - 就像文中說的，它是給「引擎 (Script/Engine)」看的，決定了 Mag 的物理極限。
        
    - **為什麼用 TOML？** 因為它比 JSON 好讀（支援註解 `#`），比 YAML 不容易寫錯縮排。
        

### 2. Dataset config toml (資料集設定)

- **您的理解：** 這是定義「資料來源」的檔案。告訴程式去哪裡抓圖片、怎麼處理資料夾。
    
- **對應到 OpenClaw (Mag)：** 這概念對應到您的 **`SKILL.md`** 和 **`MEMORY.md`**。
    
    - 在 OpenClaw 裡，我們不用 TOML 來設定資料，而是用 Markdown (`SKILL.md`) 來告訴 Mag：「去哪裡抓 YouTube (Source)、怎麼處理它 (Workflow)」。
        
    - 對於 `pkb-collect` 技能來說，`collect.py` 裡面的路徑設定就是這部分的功能。
        

### 3. Gui config json (介面設定)

- **您的理解：** 這是「使用者的存檔」。它只給圖形介面 (GUI) 看，用來記住您上次填了什麼，不會直接傳給後端去跑訓練。
    
- **對應到 OpenClaw (Mag)：** 這對應到 OpenClaw 內部的 **`sessions.json`** 或前端 Dashboard 的設定。
    
    - 當您打開 OpenClaw 的網頁後台 (Gateway UI)，您調整的視窗大小、主題顏色，通常存成 JSON。
        
    - **關鍵區別：** JSON 是給「人機介面」用的，TOML 是給「系統核心」用的。
