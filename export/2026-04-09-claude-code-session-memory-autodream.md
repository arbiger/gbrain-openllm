# Claude Code 的记忆机制到底强在哪六维记忆体系深度解析一节视频讲透指令记忆长期记忆Session Memory 与 AutoDream

大家好，欢迎回来。今天我们来聊一个特别硬核的话题—— Cloud Code 记忆系统。你有没有想过，当你跟 AI 编程助手对话的时候，它是怎么记住你说过的话、你的偏好、你项目的上下文的？更进一步，它怎么在多个会话之间保持记忆？怎么在团队成员之间同步认知？甚至它还有一个 Auto Dream 机制，能像人类做梦一样，在后台自动整理碎片化的记忆。这不是科幻，这是 Cloud Code 源码里真真切切实现了的东西。坦白说，这是我见过的 AI 编程工具中最完整的记忆架构设计。 今天这期内容，我会从源码出发，把它的六个维度、十二个核心模块全部拆开讲透。内容比较硬核，但我尽量讲的通俗，建议收藏，从头看到尾。因为后半段的几个设计真的会刷新你对AI工程的认知。在正式拆解之前，我先给大家画一个全景图。Claude Code记忆体系总共分为六个维度，这不是我随意归纳的，而是源码里每一层都有清晰的实现边界。第一层叫指令记忆。 也就是规则级记忆，它对应的就是大家可能听说过的 Claude MD 文件。这是一套多层级、优先级递增的静态指令加载系统。你可以理解为，这是你提前写好的规矩，告诉 AI 该怎么做事。第二层叫短期记忆，就是当前会话里的完整对话历史，每一条消息都存在内存里，没有压缩过的。第三层叫工作记忆，这跟当前正在执行的任务相关，比如任务的进度、偏移量、投机响应状态这些。第四层叫长期记忆。 这是最核心的部分，它基于磁盘上的 Markdown 文件构建了一个分层的知识库引擎。系统会自动在你的用户目录下创建专门的记忆文件夹，按照用户画像、行为反馈、项目上下文、外部引用这四种类型来组织知识。第五层叫摘要级，也就是压缩级记忆。当对话太长，Token 快要爆的时候，系统会触发压缩。但这里有意思的是， Claude Code 不是简单的让 LLM 生成一段摘要就完事了。 它引入了一个叫 Session Memory 的机制，通过后台子代理持续维护一份绘画笔记，压缩的时候直接用这份笔记替代临时生成的摘要。第六层叫休眠重塑记忆，也就是离线级记忆。这个名字听着就很有意思，对不对？它叫 Auto Dream 自动做梦。当满足一定时间和绘画次数的条件后，系统会自动启动一个整合流程，把碎片化的记忆清理、合并、去除矛盾，然后更新全局索引。 这六个维度层层递进，从最基础的静态规则到最高级的离线重塑，构成了一套非常完整的记忆架构。在我看来，这套设计的成熟度在当前 AI 编程助手领域里是非常领先的。它系统性的解决了三个核心问题：长程对话的上下文遗忘问题、多终端和团队之间的同步问题，以及 token 预算控制问题。好，全景图画完了，接下来我们逐一深入。先从最基础的指令记忆说起。 指令记忆的核心载体就是 cloud.md 文件组，你可以把它理解为 AI 的行为准则手册。但跟你想的不一样的是，这不是一个单一的文件，而是一套有着严格优先级体系的多层加载系统。具体来说，一共有四层优先级，从低到高分别是：第一层 manage 路径在 etc 下的 cloud code 目录里的 cloud 点 md，这是全局策略，通常由组织管理员设定，所有用户都适用。第二层 user 在你的个人目录 home 目录下的点 cloud 里的 cloud 点 md，这是你的私有全局指令。 跨所有项目生效。第三层 project 就是项目根目录下的 cloud 点 md，这个是减入代码库的团队共享的项目级指令。第四层 local cloud 点 local 点 md，这是你的私有项目指令，不会减入版本控制。这里最值得注意的是它的加载顺序，系统从 cwd，也就是当前工作目录向根目录遍历，收集所有文件，然后反转顺序，变成从根到 cwd 的顺序处理。 这意味着离你当前工作目录越近的规则文件加载越靠后，模型对它的关注度就越高。换句话说，最后加载的规则优先级最高。这个设计非常巧妙，它既保证了全局策略的基础约束力，又给了每个项目甚至每个子目录足够的灵活性。除了基本的加载逻辑，指令记忆还支持三种扩展机制。第一种是 at include 递归包含，你可以在一个 cloud 点 md 里用 at 语法引用其他文件，最多递归五层深度。 还有循环检测，防止无限递归，这样你就可以把指令模块化组织，而不是把所有规则堆在一个大文件里。第二种是条件规则，你可以在点 cloud
